# 在Anki卡片中调用AI API：从技术构想到现实挑战

## 前言

作为一个重度 Anki 用户，我一直在思考如何将 AI 的能力融入到 Anki 的学习流程中。最开始的想法很美好：在 Anki 卡片中直接调用 AI API，让 AI 帮我解释问题、生成例句、或者提供额外的知识拓展。这听起来像是一个完美的解决方案，但在实际探索过程中，我遇到了不少挑战。

## 最初的构想

### 方案一：纯前端 JavaScript

最开始想的是直接在卡片模板中写 JavaScript，使用 fetch API 调用 AI 服务。这个方案看起来最简单，但问题很多：

- **CORS 限制**：浏览器会阻止跨域请求
- **API Key 安全**：Key 直接暴露在前端，有安全风险
- **Anki 环境限制**：Anki 的 WebView 可能有各种限制
- **移动端不支持**：AnkiMobile 和 AnkiDroid 对 JS 的支持程度不同

### 方案二：Anki 插件方案

既然前端走不通，那就写一个 Anki 插件，在卡片模板中添加一些 JavaScript 代码，通过插件提供的 bridge 来调用 AI API。理论上这个方案是可行的：

1. 插件监听卡片中的事件
2. 用户点击按钮触发 API 调用
3. 插件将结果返回到卡片中显示

但很快发现了问题：

- Anki 的插件系统对网络请求的处理比较繁琐
- 需要处理各种异常情况（网络失败、API 限流等）
- 跨平台兼容性问题（桌面版、移动版）

### 方案三：使用 AnkiConnect

AnkiConnect 是一个可以通过 HTTP API 控制 Anki 的插件。我尝试过用它来实现一些功能，但很快意识到：

- AnkiConnect 主要用于与 Anki 外部交互
- 要在卡片内部调用 API，还是得写插件
- 移动版 Anki 不支持 AnkiConnect

## 现实的挑战

- **安全问题**：如果把 API Key 放在卡片模板里，那相当于把 Key 公开了。任何人只要拿到你的卡片模板，就能用你的 Key 调用 API。这绝对是不可接受的。
- **网络问题**：不是所有时候都有网络。在地铁上、飞机上，或者只是网络不好的时候，AI 功能就用不了了。而 Anki 的核心优势之一就是离线使用。
- **性能问题**：每次卡片加载或点击都要等 API 返回，会打断学习的节奏。Anki 的学习流程应该是流畅的，不应该被网络请求打断。
- **复杂度问题**：要让这个方案真正可用，需要处理 API 调用失败重试、错误提示、请求缓存、成本控制（避免意外超支）等。这已经不是一个小功能了，而是一个完整的项目。

## 最终的选择：硬编问题方案

在尝试了各种技术方案后，我意识到：**有时候最简单的方案反而是最好的**。

我最终选择了「硬编问题方案」——也就是直接把 AI 生成的内容写进 Anki 卡片里。

### 为什么选择这个方案？

1. **离线可用**：所有内容都在本地，不需要网络
2. **性能最好**：直接显示，没有任何延迟
3. **简单可靠**：不依赖任何插件或外部服务
4. **完全可控**：你可以编辑和优化 AI 生成的内容
5. **一次生成，永久使用**：虽然前期需要花时间，但长期来看是最省心的

### 工作流程

1. 在 AI 对话中生成需要的内容（解释、例句、拓展知识等）
2. 整理和编辑这些内容
3. 手动添加到 Anki 卡片中

听起来很「笨」，但实际上：

- 你可以用批量生成的方式提高效率
- 你对内容的质量有完全的掌控
- 不会遇到任何技术问题
- 可以在任何设备上正常使用

### 关于边际效应的讨论

从这个角度来看，三种方案有着不同的表现：

- **传统 Anki**：一对一模式，最容易形成键值对记忆。问题和答案固定，多次复习后容易变成机械记忆。
- **硬编码方案**：多对一模式。可以为同一个知识点准备多个不同角度的解释、例句或拓展内容。根据我的经验，3-5个不同的问题/解释就能显著避免键值对记忆，超过10个问题能带来的提升就很有限了。
- **AI 方案**：无限对一模式，也就是多对一的一个极端变体。理论上可以生成无限多的问题，但边际效应快速递减——超过10个之后，再多的新问题也不会带来更多帮助。而且技术复杂度是远超前面两个的。

## 总结

技术探索的过程很有趣，也让我学到了很多。但最终我意识到，**技术是为目的服务的**。我的目的是更好地学习，而不是追求技术上的「酷炫」。

硬编问题方案看起来不够「高级」，但它解决了所有实际问题，让我可以专注于学习本身，而不是折腾技术。

有时候，少即是多。
